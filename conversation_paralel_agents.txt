      543 +## Cambio S1 — Unificar timeouts: eliminar `responseTimeout` r
          +edundante                                                     
      544 +                                                              
      545 +**Archivo:** `SoapGatewayAdapter.java`, líneas 43-44 y 75     
      546 +                                                              
      547 +### Problema                                                  
      548 +                                                              
      549 +Hay **dos timeouts** configurados con valores diferentes:     
      550 +                                                              
      551 +```java                                                       
      552 +// Línea 43-44: timeout a nivel HTTP client (timeoutSeconds - 
          +5, mínimo 1)                                                  
      553 +HttpClient httpClient = HttpClient.create()                   
      554 +    .responseTimeout(Duration.ofSeconds(Math.max(1, properties
          +.timeoutSeconds() - 5)));                                     
      555 +                                                              
      556 +// Línea 75: timeout a nivel reactivo (timeoutSeconds completo
          +)                                                             
      557 +.timeout(Duration.ofSeconds(properties.timeoutSeconds()))     
      558 +```                                                           
      559 +                                                              
      560 +Ejemplo con `timeoutSeconds=10`: HTTP client timeoutea a los 5
          +s, pero el `.timeout()` reactivo a los 10s. El timeout del HTT
          +P client siempre gana. El `.timeout()` reactivo nunca se alcan
          +za.                                                           
      561 +                                                              
      562 +La resta `- 5` es un número mágico sin justificación. Con `tim
          +eoutSeconds ≤ 5`, `Math.max(1, ...)` fuerza 1 segundo — demasi
          +ado agresivo y no configurable.                               
      563 +                                                              
      564 +### Solución                                                  
      565 +                                                              
      566 +Eliminar `responseTimeout` del `HttpClient`. Usar **solo** el 
          +`.timeout()` reactivo, que ya cubre todo el pipeline (conexión
          + + envío + lectura de respuesta). El timeout se configura expl
          +ícitamente en un solo lugar:                                  
      567 +                                                              
      568 +```java                                                       
      569 +HttpClient httpClient = HttpClient.create();  // sin responseT
          +imeout                                                        
      570 +                                                              
      571 +this.webClient = webClientBuilder                             
      572 +    .baseUrl(properties.endpoint())                           
      573 +    .clientConnector(new ReactorClientHttpConnector(httpClient
          +))                                                            
      574 +    .build();                                                 
      575 +```                                                           
      576 +                                                              
      577 +El `.timeout(Duration.ofSeconds(properties.timeoutSeconds()))`
          + en la línea 75 se mantiene como único mecanismo de timeout.  
      578 +                                                              
      579 +Si se desea un timeout de conexión separado, debe ser una prop
          +iedad explícita (ej. `connectionTimeoutSeconds`), no una resta
          + mágica.                                                      
      580 +                                                              
      581 +### Beneficio                                                 
      582 +                                                              
      583 +- Un solo timeout que cubre todo el ciclo de vida de la reques
          +t                                                             
      584 +- Sin números mágicos                                         
      585 +- El timeout configurado es el timeout real

      636 +## Cambio S3 — Eliminar `AtomicInteger retryCount` (usar `Retr
          +y.RetrySignal`)                                               
      637 +                                                              
      638 +**Archivo:** `SoapGatewayAdapter.java`, línea 58, 80-81, 98   
      639 +                                                              
      640 +### Problema                                                  
      641 +                                                              
      642 +El `AtomicInteger retryCount` rastrea el número de reintentos 
          +manualmente, duplicando lo que Reactor ya provee. Esto introdu
          +ce **estado mutable** en una clase sin estado, con riesgos de 
          +concurrencia si el Mono se re-suscribe:                       
      643 +                                                              
      644 +```java                                                       
      645 +AtomicInteger retryCount = new AtomicInteger(0);  // mutable, 
          +compartido entre suscripciones                                
      646 +                                                              
      647 +.doBeforeRetry(retrySignal -> {                               
      648 +    int attempts = (int) retrySignal.totalRetries() + 1;      
      649 +    retryCount.set(attempts);  // duplica retrySignal.totalRet
          +ries()                                                        
      650 +})                                                            
      651 +```                                                           
      652 +                                                              
      653 +En `onErrorResume` (línea 98), `retryCount.get()` se usa para 
          +saber cuántos reintentos hubo. Pero cuando los reintentos se a
          +gotan, Reactor lanza `RetryExhaustedException` que contiene `t
          +otalRetries()`.                                               
      654 +                                                              
      655 +### Solución                                                  
      656 +                                                              
      657 +Eliminar `AtomicInteger`. Extraer el retry count de la excepci
          +ón cuando se agotaron los reintentos:                         
      658 +                                                              
      659 +```java                                                       
      660 +return soapCall                                               
      661 +    .onErrorResume(throwable -> {                             
      662 +        int retries = 0;                                      
      663 +        Throwable cause = throwable;                          
      664 +        if (Exceptions.isRetryExhausted(throwable)) {         
      665 +            cause = Exceptions.unwrap(throwable);             
      666 +            retries = (int) ((RetryExhaustedException) throwab
          +le).totalRetries();                                           
      667 +        }                                                     
      668 +        // ... usar retries                                   
      669 +    });                                                       
      670 +```                                                           
      671 +                                                              
      672 +En `doBeforeRetry` solo se loguea, sin necesidad de mutar esta
          +do:                                                           
      673 +                                                              
      674 +```java                                                       
      675 +.doBeforeRetry(retrySignal -> {                               
      676 +    long attempt = retrySignal.totalRetries() + 1;            
      677 +    log.warn("Retrying SOAP call for traceId={}, attempt {}/{}
          + (backoff={}ms)",                                             
      678 +        request.getTraceId(), attempt,                        
      679 +        properties.retryAttempts(),                           
      680 +        properties.retryBackoffMillis() * attempt);           
      681 +})                                                            
      682 +```                                                           
      683 +                                                              
      684 +### Beneficio                                                 
      685 +                                                              
      686 +- Sin estado mutable                                          
      687 +- Thread-safe sin necesidad de `AtomicInteger`                
      688 +- Una línea menos de código                                   
      689 +

      692 +## Cambio S4 — Hacer retryable `429 Too Many Requests` y error
          +es de conexión                                                
      693 +                                                              
      694 +**Archivo:** `SoapGatewayAdapter.java`, líneas 115-127        
      695 +                                                              
      696 +### Problema                                                  
      697 +                                                              
      698 +`isRetryableException` solo retrya `TimeoutException` y `5xx`.
          + No retrya:                                                   
      699 +                                                              
      700 +- **`429 Too Many Requests`**: el servidor pide backpressure e
          +xplícitamente. No reintentar es perder la oportunidad de éxito
          + con backoff.                                                 
      701 +- **Errores de conexión** (`java.net.ConnectException`, `java.
          +io.IOException`): si el socket falla al establecerse, reintent
          +ar con backoff es la estrategia correcta.                     
      702 +                                                              
      703 +```java                                                       
      704 +private boolean isRetryableException(Throwable throwable) {   
      705 +    if (throwable instanceof TimeoutException) return true;   
      706 +    if (throwable instanceof WebClientResponseException e) {  
      707 +        if (e.getStatusCode().is5xxServerError()) return true;
      708 +    }                                                         
      709 +    return false;  // 429, connection errors → no retry       
      710 +}                                                             
      711 +```                                                           
      712 +                                                              
      713 +### Solución                                                  
      714 +                                                              
      715 +Agregar `429` y excepciones de conexión como retryables:      
      716 +                                                              
      717 +```java                                                       
      718 +private boolean isRetryableException(Throwable throwable) {   
      719 +    if (throwable instanceof TimeoutException) return true;   
      720 +    if (throwable instanceof java.net.ConnectException) return
          + true;                                                        
      721 +    if (throwable instanceof java.io.IOException) return true;
      722 +    if (throwable instanceof WebClientResponseException e) {  
      723 +        HttpStatusCode status = e.getStatusCode();            
      724 +        if (status.is5xxServerError()) return true;           
      725 +        if (status.value() == 429) return true;               
      726 +    }                                                         
      727 +    return false;                                             
      728 +}                                                             
      729 +```                                                           
      730 +                                                              
      731 +### Beneficio                                                 
      732 +                                                              
      733 +- Backpressure respetada (429)                                
      734 +- Errores transitorios de red se recuperan automáticamente    
      735 +- Menos documentos en estado FAILURE por errores temporales   
      736 +

  ────────────────────────────────────────
  #: S2
  Cambio: Pasar cause a la excepción
  Problema detectado: Al crear SoapCommunicationException nunca se pasa la causa

  ────────────────────────────────────────

  Cambio: Eliminar AtomicInteger
  Problema detectado: retryCount es estado mutable en una clase sin estado.

  ────────────────────────────────────────
  #: S4

  Problema detectado: DocumentErrorCodes (clase de dominio) contiene constantes

  ────────────────────────────────────────

  Problema detectado: Faltan tests para mapeo de 4xx a CLIENT_ERROR, agotamiento

❯  Cómo funcionan las propiedades por caso de uso (SOAP vs S3)

⏺ El sistema funciona con beans separados por caso de uso inyectando settings
  distintos desde el mismo application.yml. Aquí está el flujo completo:

  Estructura en application.yml

  app:
    processors:
      soap:            # ProcessorSettings para SOAP
        max-size: 10 MB
        allowed-types: pdf,txt,csv
        folders-to-skip: []         # no excluye nada
        origin-patterns-to-send: [] # envía todo
      s3:              # ProcessorSettings para S3 (más permisivo pero con 
  filtros)
        max-size: 50 MB
        allowed-types: pdf,txt,csv,zip
        folders-to-skip: ["/tmp"]
        keywords: ["test", "mock", "archive"]
        origin-patterns-to-send: ["incoming", "documents", "archive"]
    soap:               # Config del gateway SOAP (SoapProperties)
      endpoint: ...
      timeout-seconds: 30
      retry-attempts: 3
    aws:
      s3:               # Config del gateway S3 (S3Properties)
        bucket-name: documents-bucket
        region: us-east-1

  Cómo se enlaza: DomainConfig

  ProcessorConfig (anotado con @ConfigurationProperties(prefix = 
  "app.processors")) expone getSoap() y getS3(), cada uno devolviendo un
  ProcessorSettings distinto. En DomainConfig se crean beans separados:

  // SOAP: recibe config.getSoap()
  @Bean FileValidator soapFileValidator(ProcessorConfig config) {
      return new FileValidator(config.getSoap());   // 10 MB, pdf/txt/csv
  }
  @Bean DocumentValidationRules soapDocumentValidationRules(ProcessorConfig
  config) {
      return new DocumentValidationRules(config.getSoap());  // sin filtros de 
  carpeta/origen
  }

  // S3: recibe config.getS3()
  @Bean FileValidator s3FileValidator(ProcessorConfig config) {
      return new FileValidator(config.getS3());     // 50 MB, pdf/txt/csv/zip
  }
  @Bean DocumentValidationRules s3DocumentValidationRules(ProcessorConfig
  config) {
      return new DocumentValidationRules(config.getS3());  // excluye /tmp, 
  filtra keywords
  }

  El flujo en runtime

  Cada pipeline (soapDocumentPipeline y s3DocumentPipeline) recibe su propio
  FileValidator y DocumentValidationRules por constructor. Cuando el orquestador
   SOAP procesa un documento, las reglas aplicadas son las de SOAP (10 MB, sin
  filtros). Cuando el orquestador S3 procesa, aplica las de S3 (50 MB, con
  filtros de keyword/origen).

  En resumen: no hay lógica condicional en runtime. La separación es a nivel de 
  wiring — Spring inyecta el settings correcto en cada bean, y cada pipeline
  opera con sus propias reglas sin saber nada del otro.
